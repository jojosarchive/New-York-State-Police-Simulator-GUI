27   # `pyinstaller-hooks-contrib`: The PyInstaller community hooks repository
28   
29   What happens when (your?) package doesn't work with PyInstaller? Say you have data files that you need at runtime?
30   PyInstaller doesn't bundle those. Your package requires others which PyInstaller can't see? How do you fix that?
31   
32   In summary, a "hook" file extends PyInstaller to adapt it to the special needs and methods used by a Python package.
33   The word "hook" is used for two kinds of files. A runtime hook helps the bootloader to launch an app, setting up the
34   environment. A package hook (there are several types of those) tells PyInstaller what to include in the final app -
35   such as the data files and (hidden) imports mentioned above.
36   
37   This repository is a collection of hooks for many packages, and allows PyInstaller to work with these packages
38   seamlessly.
39   
40   
41   ## Installation
42   
43   `pyinstaller-hooks-contrib` is automatically installed when you install PyInstaller, or can be installed with pip:
44   
45   ```commandline
46   pip install -U pyinstaller-hooks-contrib
47   ```
48   
49   
50   ## I can't see a hook for `a-package`
51   
52   Either `a-package` works fine without a hook, or no-one has contributed hooks.
53   If you'd like to add a hook, or view information about hooks,
54   please see below.

73   ### Setup
74   
75   [Fork this repo](https://github.com/pyinstaller/pyinstaller-hooks-contrib/fork) if you haven't already done so.
76   (If you have a fork already but its old, click the **Fetch upstream** button on your fork's homepage.)
77   Clone and `cd` inside your fork by running the following (replacing `bob-the-barnacle` with your github username):
78   
79   ```
80   git clone https://github.com/bob-the-barnacle/pyinstaller-hoooks-contrib.git
81   cd pyinstaller-hooks-contrib
82   ```
83   
84   Create a new branch for you changes (replacing `foo` with the name of the package):
85   You can name this branch whatever you like.
86   
87   ```
88   git checkout -b hook-for-foo
89   ```
90   
91   If you wish to create a virtual environment then do it now before proceeding to the next step.
92   
93   Install this repo in editable mode.
94   This will overwrite your current installation.
95   (Note that you can reverse this with `pip install --force-reinstall pyinstaller-hooks-contrib`).
96   
97   ```
98   pip install -e .
99   pip install -r requirements-test.txt
100  pip install flake8 pyinstaller
101  ```
102  
103  Note that on macOS and Linux, `pip` may by called `pip3`.
104  If you normally use `pip3` and `python3` then use `pip3` here too.
105  You may skip the 2<sup>nd</sup> line if you have no intention of providing tests (but please do provide tests!).

108  ### Add the hook
109  
110  Standard hooks live in the [src/_pyinstaller_hooks_contrib/hooks/stdhooks/](../master/src/_pyinstaller_hooks_contrib/hooks/stdhooks/) directory.
111  Runtime hooks live in the [src/_pyinstaller_hooks_contrib/hooks/rthooks/](../master/src/_pyinstaller_hooks_contrib/hooks/rthooks/) directory.
112  Simply copy your hook into there.
113  If you're unsure if your hook is a runtime hook then it almost certainly is a standard hook.
114  
115  Please annotate (with comments) anything unusual in the hook.
116  *Unusual* here is defined as any of the following:
117  
118  *   Long lists of `hiddenimport` submodules.
119      If you need lots of hidden imports then use [`collect_submodules('foo')`](https://pyinstaller.readthedocs.io/en/latest/hooks.html#PyInstaller.utils.hooks.collect_submodules).
120      For bonus points, track down why so many submodules are hidden. Typical causes are:
121      *   Lazily loaded submodules (`importlib.importmodule()` inside a module `__getattr__()`).
122      *   Dynamically loaded *backends*.
123      *   Usage of `Cython` or Python extension modules containing `import` statements.
124  *   Use of [`collect_all()`](https://pyinstaller.readthedocs.io/en/latest/hooks.html#PyInstaller.utils.hooks.collect_all).
125      This function's performance is abismal and [it is broken by
126      design](https://github.com/pyinstaller/pyinstaller/issues/6458#issuecomment-1000481631) because it confuses
127      packages with distributions.
128      Check that you really do need to collect all of submodules, data files, binaries, metadata and dependencies.
129      If you do then add a comment to say so (and if you know it - why).
130      Do not simply use `collect_all()` just to *future proof* the hook.
131  *   Any complicated `os.path` arithmetic (by which I simply mean overly complex filename manipulations).

182  ### Test
183  
184  Having tests is key to our continuous integration.
185  With them we can automatically verify that your hook works on all platforms, all Python versions and new versions of
186  libraries as and when they are released.
187  Without them, we have no idea if the hook is broken until someone finds out the hard way.
188  Please write tests!!!
189  
190  Some user interface libraries may be impossible to test without user interaction
191  or a wrapper library for some web API may require credentials (and possibly a paid subscription) to test.
192  In such cases, don't provide a test.
193  Instead explain either in the commit message or when you open your pull request why an automatic test is impractical
194  then skip on to [the next step](#run-linter).
195  
196  
197  #### Write tests(s)
198  
199  A test should be the least amount of code required to cause a breakage
200  if you do not have the hook which you are contributing.
201  For example if you are writing a hook for a library called `foo`
202  which crashes immediately under PyInstaller on `import foo` then `import foo` is your test.
203  If `import foo` works even without the hook then you will have to get a bit more creative.
204  Good sources of such minimal tests are introductory examples
205  from the documentation of whichever library you're writing a hook for.
206  Package's internal data files and hidden dependencies are prone to moving around so
207  tests should not explicitly check for presence of data files or hidden modules directly -
208  rather they should use parts of the library which are expected to use said data files or hidden modules.
209  
210  Tests currently all live in [src/_pyinstaller_hooks_contrib/tests/test_libraries.py](../master/src/_pyinstaller_hooks_contrib/tests/test_libraries.py).
211  Navigate there and add something like the following, replacing all occurrences of `foo` with the real name of the library.
212  (Note where you put it in that file doesn't matter.)
213  
214  ```python
215  @importorskip('foo')
216  def test_foo(pyi_builder):
217      pyi_builder.test_source("""
218  
219          # Your test here!
220          import foo
221  
222          foo.something_fooey()
223  
224      """)
225  ```
226  
227  If the library has changed significantly over past versions then you may need to add version constraints to the test.
228  To do that, replace the `@importorskip("foo")` with a call to `PyInstaller.utils.tests.requires()` (e.g.
229  `@requires("foo >= 1.4")`) to only run the test if the given version constraint is satisfied.
230  Note that `@importorskip` uses module names (something you'd `import`) whereas `@requires` uses distribution names
231  (something you'd `pip install`) so you'd use `@importorskip("PIL")` but `@requires("pillow")`.
232  For most packages, the distribution and packages names are the same.
233  
234  
235  #### Run the test locally
236  
237  Running our full test suite is not recommended as it will spend a very long time testing code which you have not touched.
238  Instead, run tests individually using either the `-k` option to search for test names:
239  
240  ```
241  pytest -k test_foo
242  ```
243  
244  Or using full paths:
245  ```
246  pytest src/_pyinstaller_hooks_contrib/tests/test_libraries.py::test_foo
247  ```
248  
249  
250  #### Pin the test requirement
251  
252  Get the version of the package you are working with (`pip show foo`)
253  and add it to the [requirements-test-libraries.txt](../master/requirements-test-libraries.txt) file.
254  The requirements already in there should guide you on the syntax.

257  #### Run the test on CI/CD
258  
259  <details><summary>CI/CD now triggers itself when you open a pull request.
260  These instructions for triggering jobs manually are obsolete except in rare cases.</summary>
261  
262  To test hooks on all platforms we use Github's continuous integration (CI/CD).
263  Our CI/CD is a bit unusual in that it's triggered manually and takes arguments
264  which limit which tests are run.
265  This is for the same reason we filter tests when running locally -
266  the full test suite takes ages.
267  
268  First push the changes you've made so far.
269  
270  ```commandline
271  git push --set-upstream origin hook-for-foo
272  ```
273  
274  Replace *billy-the-buffalo* with your Github username in the following url then open it.
275  It should take you to the `oneshot-test` actions workflow on your fork.
276  You may be asked if you want to enable actions on your fork - say yes.
277  ```
278  https://github.com/billy-the-buffalo/pyinstaller-hooks-contrib/actions/workflows/oneshot-test.yml
279  ```
280  
281  Find the **Run workflow** button and click on it.
282  If you can't see the button,
283  select the **Oneshot test** tab from the list of workflows on the left of the page
284  and it should appear.
285  A dialog should appear containing one drop-down menu and 5 line-edit fields.
286  This dialog is where you specify what to test and which platforms and Python versions to test on.
287  Its fields are as follows:
288  
289  1.  A branch to run from. Set this to the branch which you are using (e.g. ``hook-for-foo``),
290  2.  Which package(s) to install and their version(s).
291      Which packages to test are inferred from which packages are installed.
292      You can generally just copy your own changes to the `requirements-test-libraries.txt` file into this box.
293      * Set to `foo` to test the latest version of `foo`,
294      * Set to `foo==1.2, foo==2.3` (note the comma) to test two different versions of `foo` in separate jobs,
295      * Set to `foo bar` (note the lack of a comma) to test `foo` and `bar` in the same job,
296  3.  Which OS or OSs to run on
297      * Set to `ubuntu` to test only `ubuntu`,
298      * Set to `ubuntu, macos, windows` (order is unimportant) to test all three OSs.
299  4.  Which Python version(s) to run on
300      * Set to `3.9` to test only Python 3.9,
301      * Set to `3.8, 3.9, 3.10, 3.11` to test all currently supported version of Python.
302  5.  The final two options can generally be left alone.
303  
304  Hit the green **Run workflow** button at the bottom of the dialog, wait a few seconds then refresh the page.
305  Your workflow run should appear.
306  
307  We'll eventually want to see a build (or collection of builds) which pass on
308  all OSs and all Python versions.
309  Once you have one, hang onto its URL - you'll need it when you submit the pull request.
310  If you can't get it to work - that's fine.
311  Open a pull request as a draft, show us what you've got and we'll try and help.
312  
313  
314  #### Triggering CI/CD from a terminal
315  
316  If you find repeatedly entering the configuration into Github's **Run workflow** dialog arduous
317  then we also have a CLI script to launch it.
318  Run ``python scripts/cloud-test.py --help`` which should walk you through it.
319  You will have to enter all the details again but, thanks to the wonders of terminal history,

325  ### Run Linter
326  
327  We use `flake8` to enforce code-style.
328  `pip install flake8` if you haven't already then run it with the following.
329  
330  ```
331  flake8
332  ```
333  
334  No news is good news.
335  If it complains about your changes then do what it asks then run it again.
336  If you don't understand the errors it come up with them lookup the error code
337  in each line (a capital letter followed by a number e.g. `W391`).
338  
339  **Please do not fix flake8 issues found in parts of the repository other than the bit that you are working on.** Not only is it very boring for you, but it is harder for maintainers to
340  review your changes because so many of them are irrelevant to the hook you are adding or changing.

343  ### Add a news entry
344  
345  Please read [news/README.txt](https://github.com/pyinstaller/pyinstaller-hooks-contrib/blob/master/news/README.txt) before submitting you pull request.
346  This will require you to know the pull request number before you make the pull request.
347  You can usually guess it by adding 1 to the number of [the latest issue or pull request](https://github.com/pyinstaller/pyinstaller-hooks-contrib/issues?q=sort%3Acreated-desc).
348  Alternatively, [submit the pull request](#submit-the-pull-request) as a draft,
349  then add, commit and push the news item after you know your pull request number.

352  ### Summary
353  
354  A brief checklist for before submitting your pull request:
355  
356  * [ ] All new Python files have [the appropriate copyright header](#add-the-copyright-header).
357  * [ ] You have written a [news entry](#add-a-news-entry).
358  * [ ] Your changes [satisfy the linter](#run-linter) (run `flake8`).

362  ### Submit the pull request
363  
364  Once you've done all the above, run `git push --set-upstream origin hook-for-foo` then go ahead and create a pull request.
365  If you're stuck doing any of the above steps, create a draft pull request and explain what's wrong - we'll sort you out...
366  Feel free to copy/paste commit messages into the Github pull request title and description.
367  If you've never done a pull request before, note that you can edit it simply by running `git push` again.
368  No need to close the old one and start a new one.
