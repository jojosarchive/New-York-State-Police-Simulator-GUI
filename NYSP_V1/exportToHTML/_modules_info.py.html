12   # Qt modules information - the core of our Qt collection approach
13   # ----------------------------------------------------------------
14   #
15   # The python bindings for Qt (``PySide2``, ``PyQt5``, ``PySide6``, ``PyQt6``) consist of several python binary extension
16   # modules that provide bindings for corresponding Qt modules. For example, the ``PySide2.QtNetwork`` python extension
17   # module provides bindings for the ``QtNetwork`` Qt module from the ``qt/qtbase`` Qt repository.
18   #
19   # A Qt module can be considered as consisting of:
20   #  * a shared library (for example, on Linux, the shared library names for the ``QtNetwork`` Qt module in Qt5 and Qt6
21   #    are ``libQt5Network.so`` and ``libQt6Network.so``, respectively).
22   #  * plugins: a certain type (or class) of plugins is usually associated with a single Qt module (for example,
23   #    ``imageformats`` plugins are associated with the ``QtGui`` Qt module from the ``qt/qtbase`` Qt repository), but
24   #    additional plugins of that type may come from other Qt repositories. For example, ``imageformats/qsvg`` plugin
25   #    is provided by ``qtsvg/src/plugins/imageformats/svg`` from the ``qt/qtsvg`` repository, and ``imageformats/qpdf``
26   #    is provided by ``qtwebengine/src/pdf/plugins/imageformats/pdf`` from the ``qt/qtwebengine`` repository.
27   #  * translation files: names of translation files consist of a base name, which typically corresponds to the Qt
28   #    repository name, and language code. A single translation file usually covers all Qt modules contained within
29   #    the same repository. For example, translation files with base name ``qtbase``  contain translations for ``QtCore``,
30   #    ``QtGui``, ``QtWidgets``, ``QtNetwork``, and other Qt modules from the ``qt/qtbase`` Qt repository.
31   #
32   # The PyInstaller's built-in analysis of link-time dependencies ensures that when collecting a Qt python extension
33   # module, we automatically pick up the linked Qt shared libraries. However, collection of linked Qt shared libraries
34   # does not result in collection of plugins, nor translation files. In addition, the dependency of a Qt python extension
35   # module on other Qt python extension modules (i.e., at the bindings level) cannot be automatically determined due to
36   # PyInstaller's inability to scan imports in binary extensions.
37   #
38   # PyInstaller < 5.7 solved this problem using a dictionary that associated a Qt shared library name with python
39   # extension name, plugins, and translation files. For each hooked Qt python extension module, the hook calls a helper
40   # that analyzes the extension file for link-time dependencies, and matches those against the dictionary. Therefore,
41   # based on linked shared libraries, we could recursively infer the list of files to collect in addition to the shared
42   # libraries themselves:
43   #  - plugins and translation files belonging to Qt modules whose shared libraries we collect
44   #  - Qt python extension modules corresponding to the Qt modules that we collect
45   #
46   # The above approach ensures that even if analyzed python script contains only ``from PySide2 import QtWidgets``,
47   # we would also collect ``PySide2.QtGui`` and ``PySide2.QtCore``, as well as all corresponding Qt module files
48   # (the shared libraries, plugins, translation files). For this to work, a hook must be provided for the
49   # ``PySide2.QtWidgets`` that performs the recursive analysis of the extension module file; so to ensure that each
50   # Qt python extension module by itself ensures collection of all its dependencies, we need to hook all Qt python
51   # extension modules provided by specific python Qt bindings package.
52   #
53   # The above approach with single dictionary, however, has several limitations:
54   #  - it cannot provide association for Qt python module that binds a Qt module without a shared library (i.e., a
55   #    headers-only module, or a statically-built module). In such cases, potential plugins and translations should
56   #    be associated directly with the Qt python extension file instead of the Qt module's (non-existent) shared library.
57   #  - it cannot (directly) handle differences between Qt5 and Qt6; we had to build a second dictionary
58   #  - it cannot handle differences between the bindings themselves; for example, PyQt5 binds some Qt modules that
59   #    PySide2 does not bind. Or, the binding's Qt python extension module is named differently in PyQt and PySide
60   #    bindings (or just differently in PyQt5, while PySide2, PySide6, and PyQt6 use the same name).
61   #
62   # In order address the above shortcomings, we now store all information a list of structures that contain information
63   # for a particular Qt python extension and/or Qt module (shared library):
64   #  - python extension name (if applicable)
65   #  - Qt module name base (if applicable)
66   #  - plugins
67   #  - translation files base name
68   #  - applicable Qt version (if necessary)
69   #  - applicable Qt bindings (if necessary)
70   #
71   # This list is used to dynamically construct two dictionaries (based on the bindings name and Qt version):
72   #  - mapping python extension names to associated module information
73   #  - mapping Qt shared library names to associated module information
74   # This allows us to associate plugins and translations with either Qt python extension or with the Qt module's shared
75   # library (or both), whichever is applicable.
76   #
77   # The `qt_dynamic_dependencies_dict`_ from the original approach was constructed using several information sources, as
78   # documented `here
79   # <https://github.com/pyinstaller/pyinstaller/blob/fbf7948be85177dd44b41217e9f039e1d176de6b/PyInstaller/utils/hooks/qt.py#L266-L362>Ë™_.
80   #
81   # In the current approach, the relations stored in the `QT_MODULES_INFO`_ list were determined directly, by inspecting
82   # the Qt source code. This requires some prior knowledge of how the Qt code is organized (repositories and individual Qt
83   # modules within them), as well as some searching based on guesswork. The procedure can be outlined as follows:
84   #  * check out the `main Qt repository <git://code.qt.io/qt/qt5.git>`_. This repository contains references to all other
85   #    Qt repositories in the form of git submodules.
86   #  * for Qt5:
87   #      * check out the latest release tag, e.g., v5.15.2, then check out the submodules.
88   #      * search the Qt modules' qmake .pro files; for example, ``qtbase/src/network/network.pro`` for QtNetwork module.
89   #        The plugin types associated with the module are listed in the ``MODULE_PLUGIN_TYPES`` variable (in this case,
90   #        ``bearer``).
91   #      * all translations are gathered in ``qttranslations`` sub-module/repository, and their association with
92   #        individual repositories can be seen in ``qttranslations/translations/translations.pro``.
93   #  * for Qt6:
94   #      * check out the latest release tag, e.g., v6.3.1, then check out the submodules.
95   #      * search the Qt modules' CMake files; for example, ``qtbase/src/network/CMakeLists.txt`` for QtNetwork module.
96   #        The plugin types associated with the module are listed under ``PLUGIN_TYPES`` argument of the
97   #        ``qt_internal_add_module()`` function that defines the Qt module.
98   #
99   # The idea is to make a list of all extension modules found in a Qt bindings package, as well as all available plugin
100  # directories (which correspond to plugin types) and translation files. For each extension, identify the corresponding
101  # Qt module (shared library name) and its associated plugins and translation files. Once this is done, most of available
102  # plugins and translations in the python bindings package should have a corresponding python Qt extension module
103  # available; this gives us associations based on the python extension module names as well as based on the Qt shared
104  # library names. For any plugins and translation files remaining unassociated, identify the corresponding Qt module;
105  # this gives us associations based only on Qt shared library names. While this second group of associations are never
106  # processed directly (due to lack of corresponding python extension), they may end up being processed during the
107  # recursive dependency analysis, if the corresponding Qt shared library is linked against by some Qt python extension
108  # or another Qt shared library.
109  
110  
111  # This structure is used to define Qt module information, such as python module/extension name, Qt module (shared
112  # library) name, translation files' base names, plugins, as well as associated python bindings (which implicitly
113  # also encode major Qt version).
114  class _QtModuleDef:
115      def __init__(self, module, shared_lib=None, translations=None, plugins=None, bindings=None):
116          # Python module (extension) name without package namespace. For example, `QtCore`.
117          # Can be None if python bindings do not bind the module, but we still need to establish relationship between
118          # the Qt module (shared library) and its plugins and translations.
119          self.module = module
120          # Associated Qt module (shared library), if any. Used during recursive dependency analysis, where a python
121          # module (extension) is analyzed for linked Qt modules (shared libraries), and then their corresponding
122          # python modules (extensions) are added to hidden imports. For example, the Qt module name is `Qt5Core` or
123          # `Qt6Core`, depending on the Qt version. Can be None for python modules that are not tied to a particular
124          # Qt shared library (for example, the corresponding Qt module is headers-only) and hence they cannot be
125          # inferred from recursive link-time dependency analysis.
126          self.shared_lib = shared_lib
127          # List of base names of translation files (if any) associated with the Qt module. Multiple base names may be
128          # associated with a single module.
129          # For example, `['qt', 'qtbase']` for `QtCore` or `['qtmultimedia']` for `QtMultimedia`.
130          self.translations = translations or []
131          # List of plugins associated with the Qt module.
132          self.plugins = plugins or []
133          # List of bindings (PySide2, PyQt5, PySide6, PyQt6) that provide the python module. This allows association of
134          # plugins and translations with shared libraries even for bindings that do not provide python module binding
135          # for the Qt module.
136          self.bindings = set(bindings or [])

427  # Helpers for turning Qt namespace specifiers, such as "!PySide2" or "PyQt*", into set of applicable
428  # namespaces.
429  def process_namespace_strings(namespaces):
430      """"Process list of Qt namespace specifier strings into set of namespaces."""
431      bindings = set()
432      for namespace in namespaces:
433          bindings |= _process_namespace_string(namespace)
434      return bindings
435  
436  
437  def _process_namespace_string(namespace):
438      """Expand a Qt namespace specifier string into set of namespaces."""
439      if namespace.startswith("!"):
440          bindings = _process_namespace_string(namespace[1:])
441          return ALL_QT_BINDINGS - bindings
442      else:
443          if namespace == "PySide*":
444              return {"PySide2", "PySide6"}
445          elif namespace == "PyQt*":
446              return {"PyQt5", "PyQt6"}
447          elif namespace in ALL_QT_BINDINGS:
448              return {namespace}
449          else:
450              raise ValueError(f"Invalid Qt namespace specifier: {namespace}!")
451
